# Provenance
### *Trace why LLMs recommend what they do.*

---

## The Thesis

Search engine optimization was built on a simple premise: understand how Google ranks pages, then engineer content to rank higher. That premise held for two decades.

It is breaking now.

LLMs are increasingly the first point of recommendation for tools, products, services, and concepts — especially in technical and B2B contexts. A developer asking Claude which graph database to use, an analyst asking GPT-4 which BI tool to evaluate, an engineer asking Gemini which vector store to deploy. These are purchase-influencing moments. And unlike Google, the ranking function is opaque.

The question Provenance is built to answer:

> **Why does an LLM recommend X over Y — and what would it take to change that?**

This is not a prompt engineering problem. It is a signal intelligence problem.

---

## The Core Insight

LLM recommendations are not random and not purely memorized. They are a function of:

```
f(entity_signals, query_context, probe_context, temporal_context, demand_context)
```

Each of these is measurable. The gap between what search demand says should be recommended (Google Trends / Adwords baseline) and what an LLM actually recommends is the **divergence signal** — and that divergence is itself a feature, not noise.

Three divergence states exist:

- **LLM ahead of demand** — model recommends before search volume catches up. Early signal. Likely web-grounded on fresh authoritative content.
- **Demand ahead of LLM** — high search volume but LLM under-recommends. Content gap, authority gap, or training lag. Highest opportunity.
- **Aligned** — LLM echoes established consensus. Least analytically interesting. Useful as calibration anchor.

The goal of Provenance is to make this divergence observable, measurable, and actionable.

---

## What Provenance Is Not

- It is not a traditional SEO tool. Page rank, backlinks, and keyword density are not the primary signals.
- It is not a prompt injection or jailbreak tool. We are not trying to manipulate LLMs.
- It is not a black-box benchmark. Every signal collected has a defined collection mechanism and a clear derivation path.
- It is not a one-time audit tool. It is a tracking instrument. Signal value compounds over time.

## The Feature Extraction Model

Provenance treats LLM recommendation analysis as a **feature extraction exercise**. The LLM response is not the output — it is the raw material. From it, and from the context surrounding it, we extract a structured fingerprint.

### Signal Families

**1. Demand Signals**
Ground truth for what humans are actually searching for. Collected via Google Trends and Adwords. This is the baseline against which LLM behavior is measured — not a validation layer but a first-class feature in the fingerprint vector.

```
search_volume           # relative (0-100), pytrends
trend_velocity          # 30-day delta
related_queries         # top 5 rising
geographic_distribution # top 5 regions
```

**2. LLM Recommendation Signals**
Structured extraction from LLM probe responses. Captures not just whether an entity was recommended but how, when, in what context, and with what confidence.

```
recommendation_rank       # 1st, 2nd, mentioned, absent
mention_type              # primary / alternative / cautionary / absent
phrasing_sentiment        # positive / neutral / qualified
context_of_mention        # use-case, audience, constraint it satisfies
co_mentioned_entities     # what else appears alongside
query_sensitivity         # does rank change across query variants?
```

**3. Citation Signals**
Extracted from URLs cited by the LLM in web-grounded responses. The cited page is evidence — it tells you what content the LLM found authoritative enough to surface.

```
cited_url
domain
page_recency              # from meta/headers
content_type              # inferred: docs / blog / comparison / review / forum
entity_mention_count      # how often entity appears on cited page
```

**4. Probe Context**
Every probe is executed in a context. That context is a signal. Two identical queries with different contexts can produce different fingerprints — that variance is analytically significant.

```
# Temporal
probed_at                 # raw datetime, all temporal buckets derived

# Geographic
country
region
language
locale

# User context
user_persona              # e.g. "developer", "enterprise_buyer", "researcher"
expertise_level           # beginner / intermediate / expert
stated_use_case           # e.g. "production", "prototyping", "evaluation"

# LLM context
provider
model
temperature
system_prompt_variant

# Session
prior_context             # null or conversation history (json)
```

**5. Divergence Signals**
Computed. Never collected directly. Derived after all probes in a run complete.

```
demand_llm_alignment_score  # 0-1, how closely LLM tracks search consensus
divergence_direction         # llm_ahead / demand_ahead / aligned
cross_query_stability        # consistent recommendation across query variants?
```

## Data Philosophy

**Collect wide. Store raw. Derive late.**

No premature aggregation. No bucketing at write time. Every probe context field is a flat column on the QueryProbe table. Segmentation, cross-dimensional analysis, and persona-based aggregation are read-time operations — not write-time decisions.

This matters because:
- You do not know which dimensions will be analytically significant until you have data
- Folding variables early destroys information that cannot be recovered
- The schema should support any future aggregation without a migration

The only derived tables written at probe time are LLMSignal, Citation, and DivergenceScore. Everything else is a query.

## Execution Model

### Entity-Centric

The atomic unit of Provenance is an **Entity** — a brand, product, tool, or concept being tracked.

```
Entity {
  name:        "Kuzu"
  category:    "graph database"
  url:         "kuzudb.com"
  competitors: ["Neo4j", "ArangoDB", "TigerGraph"]
  query_seeds: ["best graph database for Python", ...]  # auto-expanded + manual
}
```

Everything runs off an Entity definition. One entity → full pipeline → fingerprint.

### Three Use Modes

**Self tracking**
How does the LLM recommend my entity? What signals is it picking up on? How has that changed over time?

**Competitor benchmarking**
Why does the LLM recommend Competitor A over me? What signals do they have that I don't? Run the same pipeline for each competitor, diff the feature vectors.

**Gap analysis**
Here is the delta. Here is what content and signals are missing. Here is what to build to close it. Prioritized by signal impact, not guesswork.

### Probe Modes

**Isolation** — single provider, single model. Pure fingerprint for that model's recommendation behavior. Useful for deep analysis of one LLM.

**Aggregate** — all configured providers. Consensus fingerprint + per-model divergence. Where models disagree is itself a signal.

### Query Variant Strategy

Every entity probe runs across four query variants to measure recommendation stability:

```
Direct       "What is the best {category}?"
Comparative  "Compare the top options for {category}. Which would you recommend?"
Expert       "As a technical expert, which {category} for a production environment?"
Contrarian   "What is the most underrated option for {category} that experts miss?"
```

Rank stability across variants = strong signal. Rank instability = LLM is uncertain = opportunity.

## What This Unlocks

### For a brand tracking itself
- Am I being recommended? At what rank? In what context?
- Is the LLM describing me accurately or with outdated signals?
- Which query framings surface me vs. bury me?
- How is my fingerprint drifting over time?

### For competitive intelligence
- What content signals does Competitor A have that cause the LLM to prefer them?
- In which persona/use-case/geographic contexts do I lead vs. lag?
- What is the minimum signal delta needed to displace the current top recommendation?

### For content strategy
- The gap analysis output is a direct content brief
- Not "write more content" but "write content that produces these specific citation signals"
- Grounded in what the LLM actually cites, not keyword density guesswork

## Architecture Principles

**API-first.** The core is a FastAPI service. Dashboard, CLI, and future agentic consumers all hit the same API. No business logic lives outside the API layer.

**Registry pattern for extensibility.** LLM probes and signal collectors are registered implementations of abstract base classes. Adding a new provider or signal source is a drop-in, not a refactor.

**Flat schema, late derivation.** Raw probe data is never aggregated at write time. All analysis is read-time computation over flat tables.

**Designed seams for v2.** The following are explicitly out of scope for v1 but the architecture must not prevent them:
- SQLite → Postgres (zero schema change, connection swap)
- FastAPI background tasks → Celery (function extraction only)
- Single-tenant → multi-tenant (API key auth layer)
- New signal collectors (registry drop-in)
- New LLM providers (registry drop-in, OpenAI and Gemini stubbed)
- Dashboard layer (consumes existing API, no new endpoints needed)
- Agentic consumption (API is already the interface)
- Segmentation and cross-dimensional analysis (read-time queries over existing flat schema)

**Migrations from day one.** Alembic. Every schema change tracked. No manual ALTER TABLE.

**Versioned API.** All routes under `/v1/`. Future breaking changes do not break existing consumers.

## Future Signal Families (Post V1)

These are known extensions. The schema and registry pattern must not foreclose them.

```
Content fingerprint signals   owned domain crawl, topical coverage, entity clarity
Authority signals             domain authority, backlink profile (Moz/Ahrefs API)
Social signals                Reddit mention tracking, HN presence, developer forum density
Temporal drift                fingerprint change across runs over time
Cross-model consensus         where all LLMs agree vs. diverge
Persona matrix                recommendation coverage across a defined persona × geo × time grid
Semantic signals              embedding-based similarity between entity description and cited content
```

## Glossary

| Term | Definition |
|------|------------|
| Entity | The brand, product, tool, or concept being tracked |
| Run | A full pipeline execution for an entity |
| QueryProbe | A single LLM call: one query variant × one model |
| ProbeContext | The full context of a QueryProbe (temporal, geographic, user, LLM) |
| Fingerprint | The assembled feature vector for an entity from a run |
| Divergence | The gap between demand baseline and LLM recommendation |
| Isolation mode | Run against a single LLM provider |
| Aggregate mode | Run across all configured providers |
| Signal family | A category of features collected by a common mechanism |
| Gap analysis | Prioritized delta between entity fingerprint and competitor or ideal fingerprint |

## The Name

**Provenance** — the origin and chain of custody of an LLM recommendation.

Not where an entity ranks. Where the recommendation *came from*. What signals created it. What content produced those signals. That chain of custody is what this tool makes visible.
